# PmergeMe (ex02) — Ford-Johnson expliqué en profondeur

Document personnel de révision. C'est l'exercice le plus dur du module : il faut implémenter le **tri par fusion-insertion de Ford-Johnson** sur **deux conteneurs** (`std::vector` et `std::deque`), tous deux différents de ceux des ex00/ex01 (`map`, `stack`). But de ce doc : pouvoir expliquer l'algorithme et défendre chaque ligne.

---

## 0. LE point à comprendre avant tout : Ford-Johnson ≠ merge sort

Beaucoup de rendus (dont une première version très courante) font un **merge sort classique avec un tri par insertion comme cas de base** : couper en deux, trier chaque moitié, fusionner. **Ce n'est PAS Ford-Johnson**, et la fiche de correction stoppe l'évaluation si l'algorithme n'est pas le bon.

Ford-Johnson (aussi appelé *merge-insertion sort*) est un algorithme précis conçu pour **minimiser le nombre de comparaisons**. Sa structure est totalement différente :

1. **Appairer** les éléments 2 par 2.
2. Dans chaque paire, identifier le **grand** et le **petit** (1 comparaison par paire).
3. **Trier récursivement les grands** (par le même algorithme).
4. Construire la « chaîne principale » (*main chain*) avec les grands triés ; chaque grand garde son petit partenaire.
5. **Insérer les petits** dans la chaîne par **insertion binaire**, dans un ordre spécial donné par les **nombres de Jacobsthal**.
6. Gérer l'élément orphelin (*straggler*) si le nombre d'éléments est impair.

La signature reconnaissable de Ford-Johnson, c'est l'étape 5 (insertion binaire en ordre de Jacobsthal). C'est ce qu'un évaluateur cherche.

---

## 1. Pourquoi ça minimise les comparaisons (l'intuition)

Trier `n` éléments demande au minimum ~`log2(n!)` comparaisons. Le tri par insertion binaire seul en fait plus que ça. Ford-Johnson s'en approche par deux idées :

- **L'appairage** fait la moitié du travail avec `n/2` comparaisons : on sait déjà, pour chaque paire, qui est le plus grand.
- **L'ordre de Jacobsthal** organise l'insertion des petits pour que, à chaque insertion binaire, la portion de chaîne à explorer ait une taille **juste sous une puissance de 2** (`2^k - 1`). Une insertion binaire dans `2^k - 1` éléments coûte exactement `k` comparaisons, sans jamais « gâcher » une comparaison. Insérer dans le désordre, au contraire, tomberait parfois sur des tailles défavorables.

Tu n'as pas besoin de prouver l'optimalité en défense, juste d'expliquer **ces deux intuitions**.

---

## 2. Les nombres de Jacobsthal

Suite définie par : `J(0)=0`, `J(1)=1`, `J(n) = J(n-1) + 2·J(n-2)`.
Ça donne : **0, 1, 1, 3, 5, 11, 21, 43, 85, 171, …**

On s'en sert pour l'**ordre d'insertion** des petits. En notant les petits `b_1, b_2, b_3, …` (où `b_i` est le partenaire du `i`-ème plus petit grand) :

- `b_1` est inséré en premier, **gratuitement**, tout au début de la chaîne (on sait que `b_1 < a_1`, le plus petit grand).
- Les suivants sont insérés par **groupes délimités par les nombres de Jacobsthal**, et **en ordre décroissant** dans chaque groupe :
  - groupe 1 : `b_3, b_2`
  - groupe 2 : `b_5, b_4`
  - groupe 3 : `b_11, b_10, b_9, b_8, b_7, b_6`
  - groupe 4 : `b_21, …, b_12`
  - etc.

L'ordre décroissant dans chaque groupe garantit que chaque `b_i` est inséré alors que la chaîne a exactement la « bonne » taille.

Dans le code, `jacobOrder(p)` génère cette suite d'indices (convertis en 0-based).

## 3. Déroulé complet sur un exemple

Trions `[3, 5, 9, 7, 4]` (n = 5, impair).

**Étape 1 — straggler** : n impair → le dernier élément `4` est mis de côté (straggler).
Reste `[3, 5, 9, 7]`.

**Étape 2 — appairage** (grand, petit) :
- paire (3,5) → (5,3)
- paire (9,7) → (9,7)
Paires = `[(5,3), (9,7)]` sous forme `(big, small)`.

**Étape 3 — trier les paires par leur grand** : les grands sont 5 et 9, déjà triés.
Paires triées = `[(5,3), (9,7)]`.
→ `main = [5, 9]` (grands triés), `pend = [3, 7]` (petits alignés).

**Étape 4 — chaîne initiale** : on met `pend[0]` (=3) devant, puis les grands.
`chain = [3, 5, 9]`.

**Étape 5 — insérer les petits restants en ordre de Jacobsthal.**
`pend = [3, 7]`, `pend[0]=3` déjà placé. Reste `pend[1]=7`.
`jacobOrder(2)` → ordre = `[1]` → on insère `pend[1]=7` par insertion binaire dans `[3,5,9]` → position entre 5 et 9.
`chain = [3, 5, 7, 9]`.

**Étape 6 — insérer le straggler** `4` par insertion binaire dans `[3,5,7,9]` → entre 3 et 5.
`chain = [3, 4, 5, 7, 9]`. **Trié.** ✓

## 4. Le header (`PmergeMe.hpp`)

```cpp
std::vector<int> _input;                       // les nombres validés, bruts (non triés)

std::vector<size_t> jacobOrder(size_t pendSize);   // ordre d'insertion (partagé)

// Ford-Johnson pour vector
void sortPairsByFirstVec(std::vector<std::pair<int,int> >& p);
void binaryInsertVec(std::vector<int>& chain, int value);
void fordJohnsonVec(std::vector<int>& v);

// Ford-Johnson pour deque (mêmes algos, types deque)
void sortPairsByFirstDeq(std::deque<std::pair<int,int> >& p);
void binaryInsertDeq(std::deque<int>& chain, int value);
void fordJohnsonDeq(std::deque<int>& v);
```

- On stocke l'entrée **une seule fois** dans `_input`. Dans `run()`, on la copiera dans un `vector` local puis un `deque` local, chacun chronométré séparément.
- Le sujet conseille : *« implement your algorithm for each container and thus avoid using a generic function »*. D'où **deux jeux de fonctions** (`…Vec` et `…Deq`), quasi identiques mais typés. `jacobOrder` reste partagé car c'est du **calcul d'indices pur**, pas le tri lui-même.
- `_input` = `std::vector<int>` : c'est juste le tampon d'entrée, pas un des deux conteneurs « de tri » à proprement parler (les tris opèrent sur un `vector` et un `deque` créés dans `run()`).

L'exception `InvalidArgumentException` hérite de `std::exception` et redéfinit `what()`. Le `throw()` après `what()` est la **spécification d'exception C++98** (« cette fonction ne lance rien ») ; en C++11 on écrirait `noexcept`, mais ici on est en C++98.

## 5. Le parsing des arguments (constructeur)

```cpp
PmergeMe::PmergeMe(int argc, char** argv) {
	for (int i = 1; i < argc; ++i) {
		std::string arg = argv[i];
		if (arg.empty())
			throw InvalidArgumentException();
		for (size_t j = 0; j < arg.length(); ++j)
			if (!std::isdigit(static_cast<unsigned char>(arg[j])))
				throw InvalidArgumentException();
		errno = 0;
		char* end = 0;
		long val = std::strtol(arg.c_str(), &end, 10);
		if (*end != '\0' || errno == ERANGE || val > INT_MAX || val < 0)
			throw InvalidArgumentException();
		_input.push_back(static_cast<int>(val));
	}
}
```

Validation en couches :
- **`arg.empty()`** : rejette un argument vide `""`.
- **Boucle `isdigit`** : chaque caractère doit être un chiffre. Ça rejette `-1` (le `-` n'est pas un chiffre), `abc`, `1.5`, etc. Comme il n'y a que des chiffres, la valeur est forcément **positive** → conforme à « positive integer sequence ». Le `static_cast<unsigned char>` évite l'UB d'`isdigit` sur un `char` négatif (même piège que dans les ex précédents).
- **`std::strtol` + contrôle d'overflow** : même après le filtre `isdigit`, une chaîne comme `9999999999` est faite de chiffres mais **dépasse `int`**. `strtol` la lit en `long` ; on vérifie `errno == ERANGE` (dépassement de `long`) et `val > INT_MAX` (dépassement d'`int`). Sans ça, la conversion en `int` déborderait silencieusement. C'est une robustesse que beaucoup de rendus oublient.

## 6. `jacobOrder` — générer l'ordre d'insertion

```cpp
std::vector<size_t> PmergeMe::jacobOrder(size_t p) {
	std::vector<size_t> order;
	if (p <= 1) return order;
	std::vector<size_t> jac;                 // bornes 1-based : 1,3,5,11,21,...
	jac.push_back(1); jac.push_back(3);
	while (jac.back() < p) {
		size_t n = jac.size();
		jac.push_back(jac[n-1] + 2 * jac[n-2]);   // récurrence de Jacobsthal
	}
	for (size_t c = 1; c < jac.size(); ++c) {
		size_t hi = jac[c];
		size_t lo = jac[c-1] + 1;
		if (hi > p) hi = p;
		if (lo > p) break;
		for (size_t j = hi; j >= lo; --j) {
			order.push_back(j - 1);          // b_j (1-based) -> index 0-based
			if (j == lo) break;              // évite le débordement de size_t
		}
	}
	return order;
}
```

- On génère les bornes de Jacobsthal `1, 3, 5, 11, 21, …` jusqu'à dépasser `p` (le nombre de petits).
- Pour chaque groupe `[jac[c-1]+1 … jac[c]]`, on pousse les indices **du plus grand au plus petit** (ordre décroissant) → c'est l'ordre `b_3,b_2` puis `b_5,b_4`, etc.
- **`j - 1`** : les `b_j` sont numérotés à partir de 1 (`b_1` est le premier petit), mais nos tableaux sont indexés à partir de 0 → on convertit.
- **`if (j == lo) break;`** : `j` est un `size_t` (non signé). Si on décrémentait en dessous de 0, il repasserait à une valeur énorme (boucle infinie). Ce garde-fou arrête la boucle proprement. **Piège classique** à savoir expliquer.

## 7. `sortPairsByFirstVec` — trier les paires par leur grand

```cpp
void PmergeMe::sortPairsByFirstVec(std::vector<std::pair<int,int> >& p) {
	size_t m = p.size();
	if (m < 2) return;

	bool odd = (m % 2 == 1);
	std::pair<int,int> straggler;
	if (odd) straggler = p[m-1];

	std::vector<std::pair<int,int> > bigs, smalls;
	for (size_t i = 0; i + 1 < m; i += 2) {
		std::pair<int,int> a = p[i], b = p[i+1];
		if (a.first < b.first) std::swap(a, b);   // a = paire au plus grand .first
		bigs.push_back(a);
		smalls.push_back(b);
	}

	sortPairsByFirstVec(bigs);                     // RÉCURSION

	std::vector<std::pair<int,int> > res = bigs;
	for (size_t i = 0; i < smalls.size(); ++i) {
		// insertion binaire (lower_bound sur .first)
		int lo = 0, hi = (int)res.size();
		while (lo < hi) {
			int mid = (lo + hi) / 2;
			if (res[mid].first < smalls[i].first) lo = mid + 1;
			else hi = mid;
		}
		res.insert(res.begin() + lo, smalls[i]);
	}
	if (odd) { /* insertion binaire du straggler, idem */ }
	p = res;
}
```

Rôle : trier un tableau de **paires** `(grand, petit)` selon leur `.first` (le grand), **sans jamais séparer** un grand de son petit. C'est la partie « trier récursivement les grands » de Ford-Johnson.

Point-clé de la conception : **les paires restent atomiques**. On déplace toujours la paire entière `(grand, petit)`. Ainsi, après le tri, chaque grand a toujours son petit partenaire collé en `.second`. C'est ce qui permet, au niveau supérieur (`fordJohnsonVec`), de récupérer les petits **alignés** avec les grands triés.

La récursion applique la même logique (appairer, récursion, insertion binaire) sur les grands eux-mêmes → c'est bien du merge-insertion, pas un merge sort.

> Note : l'ordre de Jacobsthal n'est appliqué qu'au **niveau supérieur** (dans `fordJohnsonVec`), là où l'on insère les vrais petits dans la chaîne finale. La récursion, elle, insère par recherche binaire simple — ce qui reste **correct** (le résultat est trié) ; seul le compte de comparaisons est légèrement moins optimal en profondeur. La structure reconnaissable de Ford-Johnson (appairage + tri récursif des grands + insertion binaire en ordre de Jacobsthal des petits) est bien présente.

## 8. `fordJohnsonVec` — l'orchestration

```cpp
void PmergeMe::fordJohnsonVec(std::vector<int>& v) {
	size_t n = v.size();
	if (n < 2) return;

	bool odd = (n % 2 == 1);
	int straggler = 0;
	if (odd) straggler = v[n-1];

	// 1. appairer en (grand, petit)
	std::vector<std::pair<int,int> > pairs;
	for (size_t i = 0; i + 1 < n; i += 2) {
		int a = v[i], b = v[i+1];
		if (a < b) std::swap(a, b);
		pairs.push_back(std::make_pair(a, b));
	}

	// 2. trier les paires par leur grand
	sortPairsByFirstVec(pairs);

	// 3. séparer main (grands triés) et pend (petits alignés)
	std::vector<int> main, pend;
	for (size_t i = 0; i < pairs.size(); ++i) {
		main.push_back(pairs[i].first);
		pend.push_back(pairs[i].second);
	}

	// 4. chaîne = [pend[0]] + main   (pend[0] < main[0])
	std::vector<int> chain;
	chain.push_back(pend[0]);
	for (size_t i = 0; i < main.size(); ++i) chain.push_back(main[i]);

	// 5. insérer les petits restants en ordre de Jacobsthal
	std::vector<size_t> order = jacobOrder(pend.size());
	for (size_t o = 0; o < order.size(); ++o)
		binaryInsertVec(chain, pend[order[o]]);

	// 6. insérer le straggler
	if (odd) binaryInsertVec(chain, straggler);

	v = chain;
}
```

C'est la traduction directe des 6 étapes du déroulé (section 3). `binaryInsertVec` est une insertion binaire (`lower_bound` + `insert`) qui place une valeur à sa position triée dans la chaîne.

Pourquoi `pend[0]` va-t-il tout devant sans comparaison ? Parce que `pend[0]` est le petit du plus petit grand `main[0]`, donc `pend[0] < main[0]`, qui est le minimum courant de la chaîne. Il est donc forcément le nouveau minimum.

## 9. La version `deque`

`sortPairsByFirstDeq` / `binaryInsertDeq` / `fordJohnsonDeq` sont **strictement identiques** à leurs versions `vector`, seul le **type de conteneur** change (`std::deque` au lieu de `std::vector`). Le sujet demande justement d'écrire l'algo séparément pour chaque conteneur plutôt qu'une fonction générique (template). C'est pour ça qu'on a du code « dupliqué » : c'est **voulu**.

## 10. `run` — affichage + chronométrage

```cpp
void PmergeMe::run() {
	std::cout << "Before:";
	for (size_t i = 0; i < _input.size(); ++i) std::cout << " " << _input[i];
	std::cout << std::endl;

	double t0 = getTime();
	std::vector<int> vec(_input.begin(), _input.end());   // data management (remplissage)
	fordJohnsonVec(vec);                                   // + tri
	double timeVec = getTime() - t0;

	t0 = getTime();
	std::deque<int> deq(_input.begin(), _input.end());
	fordJohnsonDeq(deq);
	double timeDeq = getTime() - t0;

	std::cout << "After: ";
	for (size_t i = 0; i < vec.size(); ++i) std::cout << " " << vec[i];
	std::cout << std::endl;

	std::cout << "Time to process a range of " << _input.size()
			  << " elements with std::vector : " << timeVec << " us" << std::endl;
	std::cout << "Time to process a range of " << _input.size()
			  << " elements with std::deque  : " << timeDeq << " us" << std::endl;
}
```

- **`getTime()`** utilise `gettimeofday` → renvoie un temps en microsecondes (`tv_sec * 1e6 + tv_usec`). On mesure `fin - début`.
- **On chronomètre le remplissage ET le tri.** Le sujet insiste : *« the time used to perform all your operations, both the sorting part and the data management part »*. En créant le `vector`/`deque` **à l'intérieur** de la zone chronométrée (à partir de `_input`), on inclut bien la « gestion des données » (allocation + copie) en plus du tri.
- **Affichage complet** de la séquence (tous les éléments). L'exemple du sujet tronque avec `[...]`, mais afficher toute la séquence est l'interprétation la plus sûre de *« display the unsorted/sorted positive integer sequence »* et permet à l'évaluateur de vérifier le tri. Si tu préfères tronquer sur les gros jeux, c'est un choix défendable aussi.

## 11. vector vs deque : pourquoi la différence de temps ?

C'est LA question de la défense (*« explain the difference in time used for each container »*).

- **`std::vector`** : mémoire **contiguë**. Accès indexé très rapide (cache-friendly). L'insertion au milieu doit décaler les éléments, mais sur un bloc contigu c'est optimisé et le CPU adore.
- **`std::deque`** : structure **segmentée** (une suite de blocs reliés). L'accès indexé passe par un niveau d'indirection, et l'insertion au milieu est plus coûteuse. Résultat : sur nos mesures, le `deque` est nettement plus lent que le `vector` pour ce motif d'accès (beaucoup d'insertions au milieu + accès aléatoires pendant les recherches binaires).

Donc : `vector` plus rapide ici parce que la **contiguïté mémoire** favorise l'accès aléatoire et les insertions par décalage, là où le `deque` paie son indirection. (Le `deque` brillerait plutôt pour des `push_front`/`push_back` intensifs, ce qu'on ne fait pas ici.)

## 12. Réponses prêtes pour la défense

- **C'est bien Ford-Johnson ?** Oui : appairage → identification grand/petit → tri récursif des grands → insertion binaire des petits en **ordre de Jacobsthal** → gestion du straggler. Ce n'est pas un merge sort (pas de « couper en deux + fusionner »).
- **À quoi servent les nombres de Jacobsthal ?** À ordonner l'insertion des petits pour que chaque insertion binaire tombe sur une portion de taille ~`2^k−1`, minimisant les comparaisons.
- **Pourquoi deux conteneurs et lesquels ?** `std::vector` et `std::deque`, différents de `map` (ex00) et `stack` (ex01). Deux implémentations séparées comme conseillé par le sujet.
- **Pourquoi vector plus rapide que deque ?** Contiguïté mémoire vs structure segmentée (voir §11).
- **Comment gères-tu l'impair ?** Un « straggler » mis de côté, réinséré à la fin par insertion binaire.
- **Overflow / entrées invalides ?** `isdigit` + `strtol` avec contrôle `ERANGE`/`INT_MAX`.
- **Fuites ?** Aucun `new`, tout en conteneurs STL (RAII). Zéro fuite.
- **Le `j == lo` break ?** `size_t` est non signé : décrémenter sous 0 boucle à l'infini, le break l'empêche.
