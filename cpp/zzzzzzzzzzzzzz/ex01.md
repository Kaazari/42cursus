# Reverse Polish Notation (ex01) — Comprendre chaque ligne en profondeur

Document personnel de révision. But : défendre **chaque ligne** de la RPN, y compris le fonctionnement de `std::stack`, `std::stringstream`, et l'algorithme d'évaluation.

---

## 0. C'est quoi la notation polonaise inverse (RPN) ?

En notation normale (infixe), l'opérateur est **entre** les opérandes : `3 + 4`.
En RPN (postfixe), l'opérateur vient **après** : `3 4 +`.

Avantage : **pas besoin de parenthèses ni de priorités**. `(1 + 2) * 3` s'écrit `1 2 + 3 *`.

L'algorithme d'évaluation tient en une phrase :
> On lit les tokens de gauche à droite. Un **nombre** → on l'empile. Un **opérateur** → on dépile les 2 derniers nombres, on calcule, on réempile le résultat. À la fin, la pile doit contenir **exactement 1** élément : le résultat.

C'est **exactement** pour ça qu'une **pile (stack)** est le bon conteneur : le dernier nombre empilé est le premier ressorti (LIFO = Last In, First Out), ce qui correspond à l'ordre naturel d'évaluation de la RPN.

Déroulé de `"3 4 + 5 *"` :
```
token "3" → push 3        pile: [3]
token "4" → push 4        pile: [3, 4]
token "+" → pop 4, pop 3, push (3+4)=7   pile: [7]
token "5" → push 5        pile: [7, 5]
token "*" → pop 5, pop 7, push (7*5)=35  pile: [35]
fin → 1 élément → résultat = 35
```

## 1. Le header (`RPN.hpp`)

```cpp
#ifndef RPN_HPP
#define RPN_HPP
```
Include guards : empêchent la double inclusion (voir doc ex00). Sans ça → note 0.

```cpp
#include <stack>       // std::stack, LE conteneur de l'exo
#include <string>      // std::string
#include <sstream>     // std::stringstream (découpage sur espaces)
#include <iostream>    // std::cout, std::cerr
#include <stdexcept>   // std::runtime_error, std::exception
#include <cstdlib>     // std::atoi
#include <cctype>      // std::isdigit
```

```cpp
class RPN {
private:
	std::stack<int> _stack;                 // la pile d'entiers

	bool isOperator(std::string const& token);
	bool isNumber(std::string const& token);
	int  calculate(int a, int b, char op);
public:
	RPN();
	RPN(RPN const& src);
	~RPN();
	RPN& operator=(RPN const& rhs);
	int evaluate(std::string const& expression);
};
```

- `std::stack<int> _stack` : une pile d'entiers. `int` car on empile des nombres et des résultats intermédiaires (qui peuvent dépasser 9, voire être négatifs).
- Les 3 helpers privés sont de la mécanique interne. Seul `evaluate` est exposé au `main`.

### Note importante sur `std::stack`

`std::stack` n'est **pas** un vrai conteneur à part entière : c'est un **adaptateur de conteneur**. Par défaut, il s'appuie sur un `std::deque<int>` en dessous, mais il n'expose qu'une interface de pile : `push`, `pop`, `top`, `size`, `empty`. Tu ne peux **pas** parcourir une stack avec un itérateur — c'est volontaire, une pile ne s'utilise que par le sommet. En défense, si on te dit "stack c'est un deque déguisé", tu réponds : oui, `std::stack` est un adaptateur au-dessus d'un `deque` (par défaut), qui restreint l'accès au seul sommet.

## 2. La Forme Canonique Orthodoxe

```cpp
RPN::RPN() {}                               // constructeur par défaut : pile vide auto

RPN::RPN(RPN const& src) {
	*this = src;                            // délègue à operator=
}

RPN& RPN::operator=(RPN const& rhs) {
	if (this != &rhs)                       // anti auto-affectation
		_stack = rhs._stack;               // std::stack sait se copier
	return *this;                          // chaînage a = b = c
}

RPN::~RPN() {}                              // rien à libérer (pas de new)
```

Même logique que dans l'ex00 (voir ce doc si besoin). Ici aussi la copie est triviale mais **obligatoire** par les règles du module.

## 3. `isOperator` — reconnaître un opérateur

```cpp
bool RPN::isOperator(std::string const& token) {
	return (token == "*" || token == "+" ||
			token == "-" || token == "/");
}
```

Renvoie `true` si le token est **exactement** l'un des 4 opérateurs autorisés. Comparaison de `std::string` avec `==` (surchargée pour comparer le contenu, pas l'adresse). Note : `"-5"` n'est PAS un opérateur (différent de `"-"`), donc un nombre négatif en entrée sera rejeté comme token invalide — cohérent avec le sujet qui ne donne que des chiffres isolés.

## 4. `isNumber` — reconnaître un nombre

```cpp
bool RPN::isNumber(std::string const& token) {
	if (token.empty())
		return false;
	if (token.length() == 1 && isdigit(static_cast<unsigned char>(token[0])))
		return true;
	return false;
}
```

- Le sujet garantit : *« The numbers ... will always be less than 10 »* → chaque nombre tient sur **un seul caractère** `0`-`9`.
- `token.length() == 1` : on exige un unique caractère. Un `"10"` (2 caractères) → refusé → sera traité comme token invalide plus loin. Défendable : le sujet ne demande pas de gérer les nombres à plusieurs chiffres en entrée.
- `isdigit(...)` : renvoie vrai si le caractère est un chiffre.
- `static_cast<unsigned char>(token[0])` : **piège C++**. `isdigit` a un comportement indéfini si on lui passe un `char` de valeur négative (caractère accentué...). On convertit d'abord en `unsigned char` (valeur garantie 0-255). Détail qui marque des points en défense.

## 5. `calculate` — appliquer un opérateur

```cpp
int RPN::calculate(int a, int b, char op) {
	switch (op) {
		case '+': return a + b;
		case '-': return a - b;
		case '*': return a * b;
		case '/':
			if (b == 0)
				throw std::runtime_error("error: division by zero");
			return a / b;
		default:
			throw std::runtime_error("error: unknown operator");
	}
}
```

- `switch (op)` sur un `char` : aiguille vers le bon calcul.
- **L'ordre `a` puis `b` est capital** : on calcule toujours `a op b`, où `a` = opérande de gauche, `b` = opérande de droite (voir section 6 pour savoir qui est qui).
- **Division par zéro** : `a / b` avec `b == 0` provoquerait un crash (SIGFPE). On teste **avant** et on lève une exception → pas de crash, message d'erreur propre. Le sujet et la fiche de correction interdisent tout crash pendant la défense.
- **Division entière** : `a / b` sur des `int` **tronque** (7 / 2 = 3, pas 3.5). Le sujet ne demande pas les décimaux, donc c'est le comportement attendu.
- `default` : sécurité si un opérateur inconnu arrivait (ne devrait jamais, car filtré par `isOperator`, mais c'est du code défensif propre).

## 6. `evaluate` — le cœur de l'algorithme

```cpp
int RPN::evaluate(std::string const& expression) {
	std::stringstream ss(expression);
	std::string token;

	while (ss >> token) {
		if (isNumber(token)) {
			int num = atoi(token.c_str());
			_stack.push(num);
		}
		else if (isOperator(token)) {
			if (_stack.size() < 2)
				throw std::runtime_error("error: insufficient operands");

			int b = _stack.top();          // 1er dépilé = opérande de DROITE
			_stack.pop();
			int a = _stack.top();          // 2e dépilé  = opérande de GAUCHE
			_stack.pop();

			_stack.push(calculate(a, b, token[0]));   // empile a op b
		}
		else {
			throw std::runtime_error("error: invalid token");
		}
	}
	if (_stack.size() != 1)
		throw std::runtime_error("error: invalid syntax");
	return _stack.top();
}
```

### Comment marche `std::stringstream` ?

- `std::stringstream ss(expression)` : crée un **flux en mémoire** initialisé avec la chaîne. C'est comme un fichier, mais qui vit en RAM.
- `ss >> token` : l'opérateur d'extraction `>>` lit **le prochain "mot"** (séquence sans espace), en **sautant automatiquement** tous les espaces/tabs/newlines avant et entre les mots. C'est ce qui gère `"  8   9 *  "` sans effort : les espaces multiples sont ignorés.
- La valeur de `ss >> token` est le flux, converti en `bool` : `true` tant qu'il a réussi à lire un mot, `false` quand il n'y a plus rien. D'où le `while (ss >> token)` qui s'arrête tout seul en fin d'expression.

### Le déroulé, token par token

1. **Nombre** → `atoi(token.c_str())` le convertit en `int` (ici un seul chiffre, donc `'7'` → `7`), puis `_stack.push(num)` l'empile.
   - `.top()` = lire le sommet **sans l'enlever**.
   - `.pop()` = **enlever** le sommet (ne renvoie rien !). D'où le pattern `top()` puis `pop()`.
2. **Opérateur** :
   - **Garde `_stack.size() < 2`** : un opérateur binaire a besoin de 2 opérandes. S'il y en a moins → expression malformée (ex : `"1 +"`).
   - **L'ordre de dépilage** : la pile est LIFO. Le **dernier** nombre empilé sort **en premier**. Dans `"7 4 -"`, on a empilé 7 puis 4 ; donc `top()` renvoie **4** d'abord → c'est `b` (droite), puis **7** → c'est `a` (gauche). On calcule `a - b = 7 - 4 = 3`. **Si on inversait, on aurait 4 - 7 = -3, faux.** C'est LA subtilité pour `-` et `/`.
   - `token[0]` : `calculate` attend un `char`, or `token` est une `std::string` d'un seul caractère → `token[0]` extrait ce `char`.
3. **Ni nombre ni opérateur** → `invalid token` (ex : `(`, `%`, `a`, `10`).

### La vérification finale

```cpp
if (_stack.size() != 1)
	throw std::runtime_error("error: invalid syntax");
```
Après avoir tout traité, une expression **valide** laisse exactement **1** résultat sur la pile.
- `> 1` restant → il y avait trop de nombres pour trop peu d'opérateurs (ex : `"1 2 3 +"` laisse 2 éléments).
- `0` restant → expression vide (`""` ou `"   "`).
Dans les deux cas → syntaxe invalide.

## 7. Le `main`

```cpp
int main(int argc, char** argv) {
	if (argc != 2) {
		std::cerr << "Usage: ./RPN \"expression\"" << std::endl;
		return 1;
	}
	try {
		RPN rpn;
		int result = rpn.evaluate(argv[1]);
		std::cout << result << std::endl;
	}
	catch (std::exception& e) {
		std::cerr << e.what() << std::endl;
		return 1;
	}
	return 0;
}
```

- `argc != 2` : on attend exactement un argument (l'expression entre guillemets). `./RPN "3 4 +"` → `argc == 2`, `argv[1] == "3 4 +"`.
- `try/catch` : toutes les exceptions levées par `evaluate`/`calculate` sont attrapées ici et affichées sur `cerr` → aucun crash, exit code 1.
- Résultat sur `cout`, erreurs sur `cerr` : séparation propre stdout/stderr.

## 8. Piège d'évaluation : le nom du binaire

Le sujet impose : *« The program name is RPN »* et montre `./RPN`.
Sur Linux, les noms de fichiers sont **sensibles à la casse** : `RPN ≠ rpn`.
Donc dans le Makefile, **`NAME = RPN`** (majuscules). Si tu mets `rpn`, l'évaluateur tape `./RPN` → *"not found"* → exercice raté alors que le code est bon. Vérifie ça avant de rendre.

## 9. Glossaire des fonctions std utilisées

| Fonction / objet | Rôle | Header |
|---|---|---|
| `std::stack<int>` | pile LIFO (adaptateur sur deque) | `<stack>` |
| `.push(x)` | empile x | `<stack>` |
| `.pop()` | retire le sommet (ne renvoie rien) | `<stack>` |
| `.top()` | lit le sommet sans le retirer | `<stack>` |
| `.size()` / `.empty()` | nb d'éléments / vide ? | `<stack>` |
| `std::stringstream` | flux en mémoire (tokenisation) | `<sstream>` |
| `ss >> token` | extrait le prochain mot, saute les espaces | `<sstream>` |
| `std::string ==` | compare le contenu de 2 chaînes | `<string>` |
| `.c_str()` | vue const char* (pont vers le C) | `<string>` |
| `std::atoi` | chaîne C → int | `<cstdlib>` |
| `std::isdigit` | caractère chiffre ? | `<cctype>` |
| `std::runtime_error` | exception avec message | `<stdexcept>` |
| `static_cast<T>` | conversion de type explicite | (langage) |

## 10. Réponses prêtes pour la défense

- **Pourquoi `std::stack` ?** Structure LIFO : parfaite pour la RPN où l'on dépile les 2 derniers opérandes à chaque opérateur. Et c'est un conteneur différent de celui de l'ex00.
- **Pourquoi `b` dépilé avant `a` ?** LIFO : le dernier empilé (opérande de droite) sort en premier. Crucial pour `-` et `/` qui ne sont pas commutatifs.
- **Comment tu évites le crash sur division par zéro ?** Test `b == 0` avant `a / b`, sinon exception.
- **Comment tu détectes une expression malformée ?** Opérateur avec < 2 opérandes → erreur ; pile ≠ 1 élément à la fin → erreur ; token non reconnu → erreur.
- **Pourquoi un seul chiffre par nombre ?** Le sujet garantit des nombres < 10. `isNumber` exige donc un unique caractère 0-9.
- **`std::stack` est-il un vrai conteneur ?** C'est un adaptateur au-dessus d'un `deque` (par défaut) ; il n'expose que le sommet.
- **Ça fuit ?** Aucun `new`, la pile gère sa mémoire en RAII. Zéro fuite.
