Document personnel de révision. But : pouvoir défendre **chaque ligne** en peer review, y compris le fonctionnement interne des outils de la STL (`std::map`, `std::ifstream`, `getline`, `strtod`...).

## 0. La vue d'ensemble : le flux de données

```
data.csv  ──loadDatabase()──►  std::map<string,float> _database  (dates triées + taux)
                                          │
input.txt ──processInput()──► pour chaque ligne :
                                 1. couper sur '|'
                                 2. valider la date      (isValidDate)
                                 3. valider la valeur     (parseValue)
                                 4. borne [0, 1000]
                                 5. getPrice(date) ──► lower_bound dans la map
                                 6. afficher  quantité * taux
```

Idée-force : la `std::map` sert **deux** besoins d'un coup — garder les dates **triées** et trouver la **date inférieure la plus proche** quand la date exacte n'existe pas. C'est *la* justification du conteneur en défense.

## 1. Le fichier header (`BitcoinExchange.hpp`)

### Les include guards

```cpp
#ifndef BITCOINEXCHANGE_HPP
#define BITCOINEXCHANGE_HPP
...
#endif
```

`#ifndef` = "if not defined". Si le symbole `BITCOINEXCHANGE_HPP` n'est pas encore défini, on le définit (`#define`) et on compile le contenu. Si un autre fichier inclut ce header une 2e fois, le symbole existe déjà → tout le bloc est sauté. **Sans ça**, inclure le header deux fois redéfinirait la classe → erreur de compilation ("double inclusion"). Le sujet dit : pas de garde d'inclusion = note 0.

### Les includes, et ce que chacun apporte

| Include | Ce qu'il fournit ici |
|---|---|
| `<map>` | `std::map`, le conteneur associatif trié |
| `<string>` | `std::string` et ses méthodes (`find`, `substr`, `size`...) |
| `<fstream>` | `std::ifstream` (lecture de fichier) |
| `<iostream>` | `std::cout`, `std::cerr`, `std::endl` |
| `<sstream>` | `std::stringstream` (pas utilisé au final, tu peux le retirer) |
| `<cstdlib>` | `std::atof`, `std::atoi`, `std::strtod` (fonctions C de conversion) |
| `<cctype>` | `std::isdigit` |
| `<stdexcept>` | `std::runtime_error`, `std::exception` (les exceptions) |

Règle du module : **un header doit inclure tout ce dont il a besoin** pour être utilisable seul.

### La structure de la classe

```cpp
class BitcoinExchange {
private:
	std::map<std::string, float> _database;   // l'attribut : clé=date, valeur=taux
	...méthodes internes...
public:
	...constructeurs/destructeur (FCO)...
	void loadDatabase(...);
	void processInput(...);
};
```

- `private` : accessible **uniquement depuis l'intérieur** de la classe. `_database` et les helpers (`trim`, `isValidDate`, `parseValue`, `getPrice`) y sont car personne d'extérieur n'a besoin d'y toucher : c'est de la mécanique interne. **Encapsulation.**
- `public` : l'interface exposée au `main`. Il n'a besoin que de 2 choses : charger la base, traiter l'input.
- `_database` : le `_` en préfixe est une convention pour "attribut membre", pas une obligation du langage.

### Déclaration vs définition

Dans le `.hpp` on **déclare** seulement les prototypes (signature + `;`). L'**implémentation** (le corps `{ ... }`) va dans le `.cpp`. Règle du module : mettre une implémentation de fonction dans un header (sauf templates) = 0. D'où la séparation stricte.

## 2. La Forme Canonique Orthodoxe (FCO)

Le module impose que toute classe ait ces **4 fonctions**. Voici pourquoi et ce que chacune fait :

```cpp
BitcoinExchange();                                        // 1. constructeur par défaut
BitcoinExchange(BitcoinExchange const& other);           // 2. constructeur par copie
BitcoinExchange& operator=(BitcoinExchange const& other);// 3. opérateur d'affectation
~BitcoinExchange();                                       // 4. destructeur
```

```cpp
BitcoinExchange::BitcoinExchange() {}          // rien à initialiser, la map se crée vide seule
BitcoinExchange::~BitcoinExchange() {}         // rien à libérer (pas de new)

BitcoinExchange::BitcoinExchange(BitcoinExchange const& other) {
	*this = other;                             // délègue à l'opérateur = pour éviter la duplication
}

BitcoinExchange& BitcoinExchange::operator=(BitcoinExchange const& other) {
	if (this != &other)                        // protection anti auto-affectation (a = a)
		_database = other._database;           // copie de la map (std::map sait se copier)
	return *this;                              // return *this permet le chaînage : a = b = c
}
```

Points de défense :
- **`const&` en paramètre** : `const` = on ne modifie pas la source ; `&` (référence) = on ne recopie pas l'objet inutilement à l'entrée.
- **`if (this != &other)`** : `this` est un pointeur sur l'objet courant. Si on fait `a = a`, inutile (et parfois dangereux) de se recopier soi-même. On compare les **adresses**.
- **`return *this`** : `this` est un pointeur, `*this` est l'objet lui-même. On le renvoie par référence pour permettre `a = b = c`.
- Ici la copie est triviale (juste une map), mais **la règle du module l'exige quand même**.

## 3. `trim` — enlever les espaces autour d'une chaîne

```cpp
std::string BitcoinExchange::trim(std::string const& str) {
	size_t first = str.find_first_not_of(" \t\n\r");
	if (first == std::string::npos)
		return "";
	size_t last = str.find_last_not_of(" \t\n\r");
	return str.substr(first, (last - first + 1));
}
```

Objectif : `"   2011-01-03   "` → `"2011-01-03"`.

- **`find_first_not_of(" \t\n\r")`** : renvoie l'index du **premier caractère qui n'est PAS** un espace/tab/newline/retour-chariot. Sur `"   abc"`, ça renvoie 3 (le `a`).
- **`std::string::npos`** : valeur spéciale "position invalide / pas trouvé" (le plus grand `size_t` possible). Si `find_first_not_of` renvoie `npos`, c'est que la chaîne ne contient **que** des espaces → on renvoie `""`.
- **`find_last_not_of(...)`** : pareil mais depuis la fin → index du dernier caractère non-espace.
- **`substr(first, length)`** : extrait une sous-chaîne à partir de l'index `first`, de longueur `last - first + 1`. Le `+1` car on veut inclure le caractère à l'index `last`.

`size_t` = type entier non signé utilisé pour les tailles/index (jamais négatif).

## 4. `isValidDate` — valider le format `YYYY-MM-DD`

```cpp
bool BitcoinExchange::isValidDate(std::string const& date) {
	if (date.size() != 10)                     // une date valide fait exactement 10 caractères
		return false;
	if (date[4] != '-' || date[7] != '-')      // tirets aux positions 4 et 7
		return false;

	for (size_t i = 0; i < date.size(); ++i) {
		if (i == 4 || i == 7)                  // on saute les 2 positions des tirets
			continue;
		if (!std::isdigit(static_cast<unsigned char>(date[i]))) // tout le reste = chiffres
			return false;
	}

	int year  = std::atoi(date.substr(0, 4).c_str());
	int month = std::atoi(date.substr(5, 2).c_str());
	int day   = std::atoi(date.substr(8, 2).c_str());

	if (year < 0)              return false;
	if (month < 1 || month > 12) return false;
	if (day < 1 || day > 31)     return false;
	return true;
}
```

- **`date.size()`** : nombre de caractères. `YYYY-MM-DD` = 4+1+2+1+2 = 10.
- **`date[i]`** : accès au i-ème caractère (comme un tableau). Positions : `Y Y Y Y - M M - D D` → index 0..3 année, 4 tiret, 5-6 mois, 7 tiret, 8-9 jour.
- **`std::isdigit(c)`** : renvoie vrai si `c` est un chiffre `'0'`..`'9'`.
- **`static_cast<unsigned char>(...)`** : **piège classique C++**. `isdigit` a un comportement **indéfini** si on lui passe un `char` de valeur négative (ce qui arrive avec les caractères accentués, car `char` peut être signé). On convertit d'abord en `unsigned char` pour garantir une valeur ∈ [0,255]. Détail qui impressionne l'évaluateur.
- **`substr(0, 4)`** : les 4 premiers caractères (année). `substr(5, 2)` : 2 caractères à partir de l'index 5 (mois). `substr(8, 2)` : le jour.
- **`.c_str()`** : `atoi` est une fonction **C** qui attend un `const char*` (chaîne C terminée par `\0`), pas un `std::string`. `.c_str()` fait le pont.
- **`std::atoi`** : convertit une chaîne C en `int`. `"07"` → `7`.

Choix assumé : je vérifie seulement `1 ≤ jour ≤ 31`, sans distinguer les mois (février à 28/29, etc.). Défendable : le sujet ne demande jamais de rejeter le 30 février.

## 5. `parseValue` — valider et convertir la valeur

```cpp
bool BitcoinExchange::parseValue(std::string const& str, float& out) {
	if (str.empty())
		return false;

	char* end = 0;
	double value = std::strtod(str.c_str(), &end);

	if (*end != '\0')     // il reste des caractères non lus => chaîne invalide
		return false;
	if (value != value)   // vrai UNIQUEMENT si value est NaN (NaN != NaN)
		return false;

	out = static_cast<float>(value);
	return true;
}
```

Le cœur, c'est **`std::strtod`** ("string to double") et son 2e paramètre :

- **`strtod(chaine, &end)`** : parse le début de `chaine` comme un nombre flottant. Le paramètre `end` (un `char**`) est **modifié par la fonction** pour pointer sur **le premier caractère qu'elle n'a pas su lire**.
  - `"12"` → `value=12`, `end` pointe sur `'\0'` (fin).
  - `"12abc"` → `value=12`, `end` pointe sur `'a'`.
  - `"1.2.3"` → `value=1.2`, `end` pointe sur le 2e `'.'`.
- **`if (*end != '\0')`** : si `end` ne pointe PAS sur la fin de chaîne, c'est qu'il restait des caractères parasites → on rejette. C'est ce qui différencie `strtod` d'`atof` : `atof` ne te dit **jamais** où il s'est arrêté, donc `atof("12abc")` renvoie 12 sans broncher.
- **`if (value != value)`** : `strtod("nan")` renvoie un NaN. Un NaN est la **seule** valeur au monde qui n'est pas égale à elle-même (propriété IEEE 754). Ce test le détecte. On l'utilise car `std::isnan` est **C++11 (interdit)**.
- **Le paramètre `float& out`** : on renvoie deux informations — un `bool` (valide ou pas) via le `return`, et la valeur numérique via la **référence** `out` (passée par l'appelant, on écrit dedans). Pattern courant quand une fonction doit renvoyer "réussite + résultat".
- **`static_cast<float>`** : `strtod` renvoie un `double` (64 bits) ; on le réduit en `float` (32 bits) car la map stocke des `float`. Le `static_cast` rend la conversion explicite (propre, pas de warning).

Pourquoi pas `std::stof` (plus simple) ? Parce que c'est **C++11, interdit**.

## 6. `loadDatabase` — remplir la map depuis `data.csv`

```cpp
void BitcoinExchange::loadDatabase(std::string const& filename) {
	std::ifstream file(filename.c_str());
	if (!file.is_open())
		throw std::runtime_error("Error: could not open database file.");

	std::string line;
	std::getline(file, line);                  // consomme la ligne d'en-tête "date,exchange_rate"

	while (std::getline(file, line)) {
		size_t comma = line.find(',');
		if (comma == std::string::npos)
			continue;
		std::string date = line.substr(0, comma);
		std::string priceStr = line.substr(comma + 1);
		float price = static_cast<float>(std::atof(priceStr.c_str()));
		_database[date] = price;               // insertion dans la map
	}

	file.close();
}
```

### Comment marche `std::ifstream` ?

- **`std::ifstream`** = "input file stream" = flux de lecture depuis un fichier. C'est un **objet** qui encapsule un fichier ouvert.
- **`std::ifstream file(filename.c_str())`** : le constructeur **ouvre** le fichier immédiatement. (En C++98, il attend un `const char*`, d'où `.c_str()`.) L'objet gère un **buffer** interne : il ne lit pas le disque caractère par caractère, il charge des blocs en mémoire pour la performance.
- **RAII** : quand `file` sort du scope (fin de la fonction), son **destructeur ferme le fichier automatiquement**. C'est pour ça qu'on n'a pas de fuite de descripteur même sans `close()`. Le `file.close()` explicite est propre mais redondant.
- **`file.is_open()`** : renvoie `true` si l'ouverture a réussi. Si le fichier n'existe pas / pas les droits → `false`. On lève alors une exception.
- **État du flux** : un stream a des drapeaux internes (`good`, `eof`, `fail`, `bad`). Dès qu'une lecture échoue (fin de fichier atteinte), le flux passe en état "faux".

### Comment marche `std::getline` ?

- **`std::getline(file, line)`** : lit dans `file` jusqu'au prochain `\n` (non inclus), stocke le résultat dans `line`, et **avance** la position de lecture.
- Sa valeur de retour est **le flux lui-même**, qui se convertit en `bool` : `true` si la lecture a réussi, `false` si on a atteint la fin du fichier. D'où le `while (std::getline(...))` : la boucle s'arrête toute seule en fin de fichier.
- Le premier `getline` **avant** la boucle sert à **jeter l'en-tête** (`date,exchange_rate`).

### Le reste

- **`line.find(',')`** : renvoie l'index de la première virgule, ou `npos` si absente.
- **`line.substr(0, comma)`** : la partie avant la virgule = la date.
- **`line.substr(comma + 1)`** : `substr` avec un seul argument = "de cet index jusqu'à la fin" = la partie après la virgule = le taux.
- **`std::atof`** : "string to float/double", convertit le taux en nombre. Ici on tolère `atof` car `data.csv` est un fichier **de confiance** (fourni par 42, bien formé) — pas besoin de la validation stricte de `parseValue`.
- **`_database[date] = price`** : l'`operator[]` de la map. S'il n'existe pas encore d'entrée pour `date`, il en **crée une** et affecte `price`. C'est la façon d'insérer/mettre à jour dans une map.

## 7. `processInput` — le cœur métier

```cpp
void BitcoinExchange::processInput(std::string const& filename) {
	std::ifstream file(filename.c_str());
	if (!file.is_open())
		throw std::runtime_error("Error: could not open file.");

	std::string line;
	std::getline(file, line);                  // jette l'en-tête "date | value"

	while (std::getline(file, line)) {
		size_t pipe = line.find('|');
		if (pipe == std::string::npos) {       // pas de séparateur => ligne invalide
			std::cerr << "Error: bad input => " << line << std::endl;
			continue;
		}

		std::string date = trim(line.substr(0, pipe));
		std::string valueStr = trim(line.substr(pipe + 1));

		if (!isValidDate(date)) {
			std::cerr << "Error: bad input => " << date << std::endl;
			continue;
		}

		float quantity;
		if (!parseValue(valueStr, quantity)) {
			std::cerr << "Error: bad input => " << valueStr << std::endl;
			continue;
		}

		if (quantity < 0) {
			std::cerr << "Error: not a positive number." << std::endl;
			continue;
		}
		if (quantity > 1000) {
			std::cerr << "Error: too large a number." << std::endl;
			continue;
		}

		try {
			float price = getPrice(date);
			float result = quantity * price;
			std::cout << date << " => " << quantity << " = " << result << std::endl;
		}
		catch (std::exception& e) {
			std::cerr << "Error: bad input => " << date << std::endl;
		}
	}

	file.close();
}
```

Points clés :

- **`std::cerr`** vs **`std::cout`** : `cerr` = flux d'**erreur** standard (stderr), `cout` = sortie standard (stdout). Les messages d'erreur vont sur `cerr` pour être séparables de la sortie normale (`./btc input.txt 2>/dev/null` masque les erreurs). C'est une bonne pratique attendue.
- **`continue`** : passe **immédiatement** à l'itération suivante de la boucle. C'est ce qui garantit l'exigence du sujet : "le programme ne doit pas s'arrêter, il traite tout le fichier". Chaque ligne fautive est signalée puis on continue.
- **L'ordre des tests est important** : d'abord le format (pipe → date → valeur numérique), **ensuite** les bornes. Ainsi `"abc"` donne `bad input` (et pas `not a positive number`), car ce n'est même pas un nombre.
- **`try / catch`** : `getPrice` peut **lever une exception** (date antérieure à toute la base). On l'attrape ici pour la transformer en `bad input` sans crasher.
- **`std::endl`** : insère `\n` **et** vide le buffer (flush). Le sujet exige un `\n` en fin de chaque message.

## 8. `getPrice` — le calcul central, et pourquoi `std::map`

```cpp
float BitcoinExchange::getPrice(std::string const& date) {
	std::map<std::string, float>::iterator it = _database.lower_bound(date);

	if (it != _database.end() && it->first == date)
		return it->second;                     // match exact

	if (it == _database.begin())
		throw std::runtime_error("Error: date too early");

	--it;                                      // recule sur la date inférieure la plus proche
	return it->second;
}
```

### Comment marche `std::map` en interne ?

- Une `std::map` est un **arbre binaire de recherche équilibré** (typiquement un **arbre rouge-noir**). Les éléments sont **triés en permanence par clé**.
- Ici les clés sont des `std::string` : la comparaison est **lexicographique** (ordre alphabétique). Et comme le format est `YYYY-MM-DD` avec zéros de remplissage, l'ordre alphabétique **coïncide avec l'ordre chronologique**. C'est LA raison pour laquelle ce format marche si bien avec une map (`"2011-01-09" < "2011-01-11"` en alphabétique = correct en dates).
- Recherche, insertion : **O(log n)** grâce à l'arbre équilibré.

### Les itérateurs

- Un **itérateur** (`std::map<...>::iterator`) est un objet qui "pointe" sur un élément de la map. On le déréférence :
  - `it->first` = la **clé** (la date),
  - `it->second` = la **valeur** (le taux).
- **`_database.end()`** : itérateur spécial "après le dernier élément" = sentinelle de "non trouvé / fin".
- **`_database.begin()`** : itérateur sur le **premier** élément (la plus petite clé).

### `lower_bound`, la pièce maîtresse

- **`lower_bound(date)`** renvoie un itérateur sur le **premier élément dont la clé est `>= date`**. Deux cas :
  1. **La date existe** → `it->first == date` → on renvoie directement `it->second`.
  2. **La date n'existe pas** → `it` pointe sur la date **juste au-dessus**. On veut la date juste **en-dessous**, donc `--it` (recule d'un cran). Sauf si `it == begin()` : alors il n'y a **aucune** date inférieure (la date demandée est plus ancienne que toute la base) → on lève une exception.
- **`--it`** décrémente l'itérateur = passe à l'élément précédent dans l'ordre trié. Possible parce que l'itérateur de map est **bidirectionnel**.

Exemple concret : la base a `2011-01-03` et `2011-01-09`. On demande `2011-01-05` (absente). `lower_bound` renvoie `2011-01-09` (première `>=`). On fait `--it` → `2011-01-03`. C'est bien la "date inférieure la plus proche" que le sujet réclame ("Be careful to use the lower date and not the upper one").

## 9. Le `main`

```cpp
int main(int argc, char** argv) {
	if (argc != 2) {
		std::cerr << "Error: could not open file." << std::endl;
		return 1;
	}
	try {
		BitcoinExchange btc;
		btc.loadDatabase("data.csv");
		btc.processInput(argv[1]);
	}
	catch (std::exception& e) {
		std::cerr << e.what() << std::endl;
		return 1;
	}
	return 0;
}
```

- **`argc`** = nombre d'arguments (le nom du programme compte pour 1). `./btc input.txt` → `argc == 2`. Sans argument → `argc == 1` → message + `return 1`.
- **`argv`** = tableau des arguments. `argv[0]` = `"./btc"`, `argv[1]` = `"input.txt"`.
- **`try/catch` global** : attrape toute exception remontée (fichier introuvable...). `e.what()` renvoie le message de l'exception.
- **`return 1`** = code de sortie "erreur" ; `return 0` = succès. Convention Unix.
- Attention : `"data.csv"` est **codé en dur** → lancer `./btc` depuis le dossier contenant `data.csv`.

## 11. Réponses prêtes pour la défense

- **Pourquoi `std::map` ?** Conteneur trié par clé ; le format `YYYY-MM-DD` rend l'ordre alphabétique = chronologique ; `lower_bound` donne la date inférieure la plus proche en O(log n). Ni `vector` ni `list` n'offrent ça gratuitement.
- **Comment tu gères une date absente ?** `lower_bound` + `--it`, avec garde `begin()` pour les dates trop anciennes.
- **Pourquoi `strtod` et pas `stof` / `atof` ?** `stof` = C++11 interdit ; `atof` ne détecte pas les résidus (`12abc`→12). `strtod` + `endptr` valide toute la chaîne. Le `value != value` bloque même les NaN.
- **Ça fuit ?** Aucun `new` ; la map gère sa mémoire en RAII ; le flux se ferme via son destructeur. Zéro fuite.
- **Le `static_cast<unsigned char>` devant `isdigit` ?** Évite l'UB d'`isdigit` sur un `char` négatif.
- **Copie de l'objet ?** FCO complète, la map est copiée, protection anti auto-affectation.
